What Is a Portfolio Rationalization Review?

  • A structured initiative to evaluate legacy SharePoint solutions and determine their appropriate future state
  • Not merely a technical audit — it is a governance exercise
  • Applicable to any organization, regardless of portfolio size

Why It Matters

  • Reduces organizational risk from aging, undocumented tools
  • Eliminates redundant maintenance overhead
  • Replaces one-off builds with modern, repeatable patterns
  • Establishes a supportable, documented operational baseline

Two Parallel Objectives

Obj 1: Strategic Disposition Obj 2: Operational Documentation
  • Obj 1: Consult enterprise architecture → assign sunset / migrate / modernize / retain
  • Obj 2: Document each solution so support requests can be triaged without escalating to a developer

Strategic Alignment

  • Does each solution still serve an active, clearly defined business need?
  • Has the need evolved — does a better tool now exist?
  • Does retaining this create shadow IT or governance debt?

Total Cost of Ownership

  • Ongoing maintenance burden (licensing, dev time, support tickets)?
  • Risk cost of keeping it undocumented and aging?
  • Cost of migration vs. cost of continued maintenance?

Risk & Compliance

  • Sensitive or regulated data? Compliant with current standards?
  • Dependent on deprecated tech or unsupported third-party components?
  • Business impact if this solution breaks today with no documentation?

Standardization & Scalability

  • One-off build or a pattern replicated across the organization?
  • Can it be replaced by a repeatable, low-complexity enterprise pattern?
  • Opportunity to consolidate multiple similar solutions?

Developer & Support Dependency

  • Is institutional knowledge locked in a single person?
  • Can it be triaged without deep SharePoint expertise?
  • How hand-off ready is this solution today?

Stakeholder & Change Management

  • Who is the current business owner? Are they actively engaged?
  • Will sunsetting or migrating create stakeholder friction?
  • Is executive sponsorship available for disposition decisions?

🔴 Sunset▶
Use when: The underlying business need no longer exists.

Ask: Has the process, policy, or regulation this supported been retired? Is there zero active usage? Are there no foreseeable future use cases?

Key actions: Archive data per retention policy, notify stakeholders, decommission the solution.
🟡 Migrate▶
Use when: The business need is active and an existing enterprise platform already meets it.

Ask: Does an approved enterprise tool already do this? Can users be moved with minimal disruption?

Key actions: Map current functionality to the target platform, plan the migration, retire the legacy solution.
🔵 Modernize▶
Use when: The business need is active but no enterprise tool is a perfect fit — however, one could be adapted or a low-complexity pattern applied.

Ask: What is the rebuild effort vs. the ongoing maintenance debt? Is there a standardized low-complexity pattern (e.g., mDFFS) that fits?

Key actions: Scope the modernization effort, select the target pattern, build the business case.
🟢 Retain & Document▶
Use when: The solution remains the best fit and is actively used.

Ask: Is it stable? Is it supportable without deep expertise? Is it documented well enough to triage?

Key actions: Complete Objective 2 operational documentation, assign triage ownership, schedule periodic review.

Resolve at First-Line Support

  • Add or remove a user from a SharePoint group
  • Direct user to the appropriate entitlements or access request platform
  • Make a simple DFFS form configuration change
  • Update a list view or column setting
  • Provide user education on how to use the solution

Escalate to Developer

  • DFFS rules logic is broken or needs redesign
  • Workflow or Power Automate flow failure
  • Data integrity issue or missing records
  • Custom code (JavaScript, REST API) is broken
  • Complex SharePoint permissions inheritance issue

First-Response Triage Questions

  • Which solution is affected? (Get the URL)
  • What were you trying to do when the issue occurred?
  • When did it last work correctly?
  • Has anything changed recently? (permissions, migration, platform update)
  • How many users are affected?

Low-Complexity Patterns: The mDFFS Example

  • A key opportunity in any rationalization review is identifying where a Standardized Low-Complexity Pattern can replace multiple one-off builds
  • Modern DFFS (mDFFS) is one such pattern for form-centric SharePoint solutions
  • An evidence-based case must be built through real data gathered during the review

Data to Capture During the Review

  • Developer hours required per modernization effort
  • Licensing costs avoided vs. alternative platforms
  • Support ticket volume before and after modernization
  • Business owner satisfaction with rebuilt solutions
  • Use this data to present a formal pattern proposal to enterprise architecture
